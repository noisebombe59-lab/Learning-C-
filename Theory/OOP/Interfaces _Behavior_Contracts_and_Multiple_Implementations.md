Interfaces (Контракты)
Суть: «Когда мне не важно, как работает объект, важно лишь, что он умеет делать (например, "мне всё равно, как ты сохраняешь данные, просто реализуй метод Save()")».

1. Явная реализация + Полиморфизм (Ролевая модель)
Суть: Способ сделать класс «скромным». Метод принадлежит не объекту напрямую, а только его конкретной «роли» (интерфейсу). Решает конфликты имен.
        
        C#
        public interface ICleaner { void Clean(); }
        public interface IManager { void Clean(); } // Одинаковые имена!
        
        public class UniversalRobot : ICleaner, IManager
        {
            // Явная реализация: доступна только через приведение к интерфейсу
            void ICleaner.Clean() => Console.WriteLine("Робот чистит пол");
            void IManager.Clean() => Console.WriteLine("Робот чистит кэш данных");
        }
        
        // Использование:
        UniversalRobot bot = new();
        // bot.Clean(); // Ошибка! Объект не знает, какую роль выбрать.
        ((ICleaner)bot).Clean(); // Вызвали роль уборщика
Когда нужен: Когда класс выполняет несколько разных задач и нужно скрыть "технические" методы, чтобы не загромождать основной список команд объекта.

2. IEnumerable + Фильтрация (Защита инкапсуляции)
Суть: Архитектурный стандарт «одностороннего шлюза». Ты отдаешь данные наружу для чтения (и для LINQ!), но запрещаешь их изменять.

        C#
        public class Vault
        {
            private List<string> _secrets = new() { "Код 1", "Код 2" };
        
            // Отдаем интерфейс для чтения, а не сам список
            public IEnumerable<string> GetSecrets() => _secrets; 
        }
        
        var vault = new Vault();
        var data = vault.GetSecrets(); 
        // data.Add("Взлом"); // Ошибка! У IEnumerable нет метода Add.
        var check = data.Any(s => s.Contains("1")); // LINQ работает идеально
Когда нужен: Всегда, когда ты выставляешь коллекцию наружу из класса. Это страховка от того, что другой программист случайно не удалит данные из твоего приватного списка.

3. DI + Интерфейсы (Гибкость системы)
Суть: Метод просит минимально необходимый «навык», а не конкретного исполнителя. Код зависит от абстракции, а не от деталей.

        C#
        public interface ILogger { void Log(string msg); }
        
        public class FileLogger : ILogger { public void Log(string msg) => /*...*/; }
        public class CloudLogger : ILogger { public void Log(string msg) => /*...*/; }
        
        public class OrderService
        {
            // Мы не знаем, КУДА пишется лог. Нам прислали "что-то", что умеет Log()
            public void PlaceOrder(ILogger logger) 
            {
                // ...логика...
                logger.Log("Заказ создан"); 
            }
        }
Когда нужен: Чтобы при смене способа хранения данных или отправки уведомлений тебе не пришлось переписывать всю бизнес-логику программы.

4. Методы по умолчанию (Default Interface Methods)
Суть: Способ добавить новую «кнопку» в интерфейс, не заставляя всех владельцев старых классов переписывать свой код.

        C#
        public interface INotifier
        {
            void Send(string message);
            
            // Новый метод с готовым кодом. Старые классы не сломаются!
            void SendUrgent(string message) => Send("СРОЧНО: " + message);
        }
        
        public class EmailNotifier : INotifier
        {
            public void Send(string msg) => Console.WriteLine(msg);
            // SendUrgent можно использовать сразу, ничего не дописывая.
        }
Когда нужен: При обновлении больших систем или библиотек, когда нужно расширить функционал интерфейса, не «сломав» десятки уже написанных классов.

🛠 Итоговая шпаргалка архитектора:
Нужен доступ только для чтения? — Используй IEnumerable<T>.

Хочешь скрыть метод от «лишних глаз»? — Используй Явную реализацию.

Хочешь, чтобы код не зависел от деталей? — Используй интерфейс в параметрах (DI).

Нужно добавить общую логику для всех сразу? — Используй Default Methods.
