краткая теория по основам ООП

1. Класс vs Объект
Класс — это «чертеж» или тип данных. Он описывает, что объект будет знать (поля) и что он будет уметь (методы).

Объект — это конкретный «экземпляр», созданный по этому чертежу в памяти компьютера через new.

2. Конструктор
Специальный метод, который «собирает» объект при создании:

Назначение: Гарантировать, что объект сразу получит необходимые данные.

Перегрузка: В классе может быть несколько конструкторов (например, пустой и с параметрами).

3. Свойства и Поля (Инкапсуляция)
Это механизм защиты данных объекта от внешнего вмешательства:

Поля (private): Скрытые внутренние переменные. Прямой доступ к ним извне запрещен.

Свойства (public): «Умные» обертки над полями.

get — позволяет безопасно прочитать значение.

set — позволяет проверить данные (валидация) перед записью.

Автосвойства: Краткая запись public string Name { get; set; }, если сложная логика проверки пока не нужна.

4. Ключевое слово this
Указывает на текущий экземпляр класса.

Зачем: Чтобы компьютер не путался, где поле класса, а где — входящий параметр метода, если у них одинаковые имена.

Пример: this.Name = name; (this.Name — свойство объекта, name — переменная из скобок).

5. Почему это важно
Переход к классам — это переход от «просто кода» к моделированию реальности.
Вместо разрозненных переменных мы создаем самостоятельные сущности, которые сами отвечают за свою корректность. 
Это фундамент, на котором строятся все крупные приложения.

Краткий свод правил качественного проектирования (Clean Code & Architecture):

1. Эшелонированная валидация
Не доверяйте входным данным. Проверяйте их дважды:

На входе (UI): Проверка формата (число ли это? не пустая ли строка?). Это нужно для удобства пользователя.

Внутри (Domain): Проверка смысла (может ли возраст быть -5?). Это нужно для безопасности системы.

2. Принцип единственной ответственности (SRP)
Разделяйте «мозги», «память» и «голос» программы:

Сущность: Хранит данные о себе.

Сервис: Управляет логикой и списками.

UI: Только общается с пользователем (никаких расчетов внутри Console.WriteLine).

3. Защита состояния (Encapsulation)
Объект не должен быть «проходным двором»:

Иммутабельность: Важные данные (ID, автор) закрывайте для изменений после создания.

Приватные сеттеры: Изменяйте состояние только через специальные методы. Это гарантирует, что данные не будут испорчены случайно извне.

4. Безопасность данных и коллекций
Копии вместо оригиналов: Выдавайте наружу копию списка. Так никто не сможет удалить данные из вашего сервиса в обход правил.

Нет значению null: Вместо null возвращайте пустые списки или используйте паттерн TryDoSomething. Это защитит программу от внезапных падений.

5. Поток управления и UX
Программа должна быть предсказуемой:

Циклы ввода: Не вылетайте с ошибкой, а вежливо просите повторить ввод.

Передача параметров: Помните, что простые типы передаются по значению (копируются), а объекты — по ссылке.

Паузы: Дайте пользователю прочитать сообщение о результате, прежде чем двигаться дальше.
