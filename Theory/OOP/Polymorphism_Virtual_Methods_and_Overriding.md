Polymorphism (Многоформенность)
Суть: «Когда я хочу вызвать один метод, но получить разный результат в зависимости от того, какой объект перед нами (Кнопка "Печать" печатает чек для клиента и отчет для админа по-разному)».

1. Механика Virtual & Override (Гибкое поведение)
Суть: Позволяет базовому классу определить «стандарт» (разрешение на тюнинг), а наследникам — создать свою «прошивку» (новую версию алгоритма), сохраняя единое имя метода.

        C#
        public class Logger {
            // virtual — "Я разрешаю наследникам менять этот метод"
            public virtual void Log(string message) => Console.WriteLine($"[LOG]: {message}");
        }
        
        public class FileLogger : Logger {
            // override — "Я заменяю стандартный метод своей логикой"
            public override void Log(string message) {
                System.IO.File.AppendAllText("log.txt", message);
            }
        }
Когда нужен: Когда у тебя есть группа похожих объектов (Грузовик, Легковушка, Мотоцикл) и каждый должен по-своему выполнять общую команду (например, CalculateTax()).

2. Приведение типов (Casting)
Суть: Возможность смотреть на сложный объект через призму его базовой роли.

Upcasting (Вверх): К родителю. Автоматически и безопасно. Мы «забываем» детали, видим только общую роль.

Downcasting (Вниз): К наследнику. Опасно. Нужна проверка, есть ли у объекта эти детали.

    C#
    // Безопасная проверка и приведение (is)
    if (logger is FileLogger fl) {
        // Теперь мы точно знаем, что это FileLogger, и работаем с fl
    }
    
    // Попытка приведения (as)
    var fileLog = logger as FileLogger; // Если не выйдет — даст null
Когда нужен: Когда у тебя есть общий список List<Vehicle>, но для конкретной машины тебе нужно вызвать уникальный метод, которого нет у других (например, OpenTrunk()).

3. Скрытие метода (New) — «Обрыв связи»
Суть: Создание метода с тем же именем, который не участвует в полиморфизме. Это не замена старого метода, а создание «дубликата», который виден только если переменная имеет тип наследника.

        C#
        public class SecretFileLogger : Logger {
            public new void Log(string msg) => /* Скрытая логика */ ;
        }
        Разница: * Если метод override — всегда сработает логика наследника.
        
        Если метод new — сработает метод того типа, который указан у переменной (Logger вызовет старый метод, SecretFileLogger — новый).
Когда нужен: Крайне редко. В 99% случаев для твоих задач нужен именно override.

🛠 Золотые правила архитектора:
Принцип «Единого пульта»: Сложи разные объекты в один List<Base> и вызывай virtual метод в цикле. Каждый объект сам знает, как себя вести.

Защита через is: Никогда не делай жесткое приведение (Circle)shape «в лоб». Всегда проверяй: if (shape is Circle c).

Не ломай ожидания: Если override метод называется CalculateArea, он должен считать площадь, а не отправлять SMS.
