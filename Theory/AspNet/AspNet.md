200 OK (GET/PUT) — данные успешно переданы или обновлены. return Ok(data);

201 Created (POST) — объект создан, в ответе есть ссылка на него. return CreatedAtAction(...);

204 No Content (DELETE) — удаление прошло успешно, показывать нечего. return NoContent();

400 Bad Request — клиент прислал некорректные данные. return BadRequest();

404 Not Found — объект с таким ID в базе не обнаружен. return NotFound();

500 Internal Server Error — на сервере произошла непредвиденная ошибка (баг).

1. API (Интерфейс)
Что это: «Стойка регистрации» твоего приложения. Фронтенд не лезет в твой код, он просто стучится в API.

Зачем нужно: Чтобы разные программы (сайт, мобильное приложение) могли безопасно общаться с твоей базой данных.

Как делать: Создавать контроллеры, которые принимают запросы и возвращают данные в формате JSON.

Нюанс: API — это не вся программа, а только её «внешняя оболочка».

2. Маршрутизация (Routing)
Что это: Система навигации. Она решает, какой метод в C# запустить, когда пришел запрос на конкретный URL.

Зачем нужно: Без неё сервер не поймет, что делать с адресом api/stock/5.

Как делать: Использовать атрибуты [Route] над контроллером и [HttpGet/HttpPost] над методами.

Нюанс: Порядок регистрации Middleware в файле Program.cs критичен для работы маршрутов.

3. Endpoint (Эндпоинт)
Что это: Конкретная «дверь» в твоем API.

Зачем нужно: Эндпоинт соединяет адрес (Куда?) и метод (Что сделать?).

Формула: URL + HTTP Глагол (например, GET + /api/stock).

Нюанс: На один адрес можно повесить разные эндпоинты (один выдает список, другой создает новую запись).

Раздел 2: Безопасность и Инструменты
4. DTO (Объект передачи данных)
Что это: Упрощенная копия твоей модели (посылка).

Зачем нужно: Чтобы не отдавать лишние данные (пароли, личные ID) и не давать хакерам менять то, что им не положено (Over-posting).

Как делать: Создаешь отдельный класс с нужными полями и пишешь «Маппер» (метод расширения), который перекладывает данные из модели в DTO.

Нюанс: Лучше иметь разные DTO для создания (CreateDto) и для ответа (ResponseDto).

C#
// Маппер превращает тяжелую модель БД в легкий DTO
public static StockDto ToStockDto(this Stock stockModel) {
    return new StockDto { Symbol = stockModel.Symbol };
}
5. DI Lifetimes (Сроки жизни объектов)
Что это: Настройки того, как долго живет объект (например, соединение с базой).

Зачем нужно: Чтобы не тратить память впустую и чтобы данные внутри одного запроса не путались.

Как делать: Регистрировать в Program.cs:

Transient — каждый раз новый (быстрые задачи).

Scoped — один на один запрос клиента (самый важный для БД).

Singleton — один на весь сервер (кэш).

Раздел 3: CRUD (Действия с базой)
6. POST (Создание) и CreatedAtAction
Что это: Создание новой записи и правильный ответ сервера.

Зачем нужно: По стандарту REST после создания нужно вернуть статус 201 и ссылку, где теперь лежит этот объект.

Как делать: Используем CreatedAtAction, указывая имя метода GET и ID нового объекта.

C#
[HttpPost]
public IActionResult Create([FromBody] CreateStockDto dto) {
    var model = dto.ToModel(); // Превратили DTO в Модель
    _context.Stocks.Add(model); // Сказали базе "Добавь"
    _context.SaveChanges();     // Нажали "Сохранить"
    return CreatedAtAction(nameof(GetById), new { id = model.Id }, model.ToDto());
}
7. PUT (Обновление)
Что это: Изменение существующей записи.

Зачем нужно: Чтобы изменить данные (цену, название), не создавая новую строку в базе.

Как делать: Обязательно сначала найти оригинал в базе по ID, изменить его свойства и сохранить.

Нюанс: Если не найти оригинал, а создать новый объект — база его не узнает.

C#
[HttpPut("{id}")]
public IActionResult Update([FromRoute] int id, [FromBody] UpdateDto dto) {
    var stock = _context.Stocks.FirstOrDefault(x => x.Id == id); // 1. Ищем
    if (stock == null) return NotFound();
    stock.Symbol = dto.Symbol; // 2. Меняем оригинал
    _context.SaveChanges();    // 3. Сохраняем
    return Ok(stock.ToDto());
}
8. DELETE (Удаление)
Что это: Полное удаление записи.

Зачем нужно: Очистка базы от ненужных данных.

Как делать: Найти по ID -> Remove -> SaveChanges.

Нюанс: Возвращаем статус 204 NoContent, так как объекта больше нет и показывать нечего.

C#
[HttpDelete("{id}")]
public IActionResult Delete([FromRoute] int id) {
    var stock = _context.Stocks.FirstOrDefault(x => x.Id == id);
    if (stock == null) return NotFound();
    _context.Stocks.Remove(stock);
    _context.SaveChanges();
    return NoContent();
}

Раздел 4: Результаты и Чтение
9. GET (Чтение)
Что это: Самый частый метод в интернете. Используется только для получения данных без их изменения.

Зачем нужно: Позволяет фронтенду запрашивать информацию: или список всех акций, или одну конкретную по её ID.

Как делать: * Для списка: _context.Stocks.ToList().

Для одного объекта: _context.Stocks.Find(id) или FirstOrDefault.

Нюанс: GET-запросы кэшируются браузерами, поэтому они должны быть «безопасными» (не менять ничего в базе).

C#
[HttpGet]
public IActionResult GetAll() {
    var stocks = _context.Stocks.ToList(); // Достаем всё
    return Ok(stocks.Select(s => s.ToStockDto())); // Превращаем в DTO и отдаем
}
10. IActionResult (Гибкий ответ)
Что это: Универсальный тип возвращаемого значения для методов контроллера.

Зачем нужно: Твой метод может закончиться по-разному. Например, если акция найдена — ты возвращаешь Ok() (статус 200), а если нет — NotFound() (статус 404). IActionResult позволяет возвращать и то, и другое из одного метода.

Как делать: Всегда указывай его как тип возврата в методах контроллера.

Нюанс: Он сам упаковывает твои объекты в JSON и проставляет нужные HTTP-статусы.

C#
public IActionResult GetSomething(int id) {
    if (id < 0) return BadRequest("ID не может быть отрицательным"); // Статус 400
    return Ok("Все отлично!"); // Статус 200
}
11. CreatedAtAction (Вежливый ответ)
Что это: Специальный метод для ответа на POST запрос (создание).

Зачем нужно: По правилам REST, когда ты что-то создал, ты должен не просто сказать «готово», а:

Вернуть статус 201 Created.

Прислать в заголовке ответа прямую ссылку на новую запись.

Как делать: Передаешь имя метода (обычно это GetById), параметры (ID новой записи) и сам объект в формате DTO.

Нюанс: Он делает твой API профессиональным и удобным для фронтенд-разработчиков.

C#
// Внутри метода POST:
return CreatedAtAction(
    nameof(GetById),         // К какому методу обращаться для просмотра?
    new { id = model.Id },   // С каким ID?
    model.ToStockDto()       // Что показать в теле ответа?
);

Раздел 5: Системные механизмы
12. Middleware (Конвейер обработки)
Что это: Представь заводской конвейер. Запрос клиента — это деталь, которая едет по ленте, а Middleware — это станки вдоль этой ленты.

Зачем нужно: Чтобы каждый запрос проходил через цепочку проверок (авторизация, логирование, обработка ошибок) ДО того, как он попадет в твой контроллер.

Как делать: Они регистрируются в файле Program.cs через методы app.Use....

Нюанс: Порядок имеет значение! Если ты поставишь проверку авторизации ПОСЛЕ маршрутизации, система может сначала найти метод, а потом уже думать, пускать туда пользователя или нет.

C#
var app = builder.Build();

app.UseHttpsRedirection(); // Станок 1: Перенаправляет на безопасный протокол
app.UseRouting();          // Станок 2: Ищет, какой контроллер нужен
app.UseAuthorization();    // Станок 3: Проверяет права доступа
app.MapControllers();      // Конец ленты: Отдает запрос в руки контроллера
13. [ApiController] (Умный помощник)
Что это: Специальный атрибут над классом контроллера, который включает «режим суперсилы» для API.

Зачем нужно: Он берет на себя нудную рутину, которую раньше программисты писали руками.

Что он делает:

Авто-валидация: Если клиент прислал текст вместо числа, он сам вернет ошибку 400 Bad Request (тебе не нужно писать if (!ModelState.IsValid)).

Умный поиск данных: Он сам понимает, что DTO нужно искать в теле запроса (FromBody), а ID — в ссылке (FromRoute).

Требует маршрутизацию: Он заставляет тебя обязательно указывать [Route], чтобы API было предсказуемым.

Нюанс: Без этого атрибута твой контроллер будет вести себя как старый сайт из 2010-х, и тебе придется вручную обрабатывать каждую ошибку ввода.

C#
[ApiController] // Включаем "автопилот" для API
[Route("api/stock")]
public class StockController : ControllerBase 
{
    // Благодаря [ApiController], ASP.NET сам поймет, 
    // что stockDto надо достать из JSON в теле запроса.
    [HttpPost]
    public IActionResult Create(CreateStockDto stockDto) 
    {
        return Ok();
    }
}
